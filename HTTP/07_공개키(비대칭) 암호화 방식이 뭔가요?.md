# 🔐 암호화와 복호화에 대해 짚고 넘어가자

데이터를 안전하게 전달하기 위해 **암호화**, **복호화** 개념이 적용된다.

`a`라는 데이터를 전달하는데 그냥 전달하면 중간에 슬쩍 보면 `a`가 보이니까 우리는 이 `a`를 `"ㅓ2304ㅜ23ㅐㅓㅜ3"` 와 같은 **암호문**으로 바꿔서 보낼 수 있는데 이것이 **암호화**이다.

`a` 데이터를 받는 이는 당연히 해당 암호문을 `a`로 해석할 수 있어야 하는데 이 과정이 **복호화**이다.

---

🔑 공개 키 암호화 방식을 알기 전, 더 쉬운 **비공개 키(대칭)** 암호화 방식부터 알아보자.

# 🔒 비공개 키(대칭) 암호화 방식 (private-key cryptography)

비공개 키(= 대칭 키) 암호화 방식은 **암호화**, **복호화**에 **똑같은 하나의 키**를 사용하는 방식이다.

```
a -> ㅓ2304ㅜ23ㅐㅓㅜ3 -> a
암호화              복호화  
(같은 키 사용)
```

⚡ 속도가 빠르고 구현이 간단해서 **실제 데이터 전송에 많이 사용**되지만,  
이 키를 **교환하는 과정이 어렵고 위험**하다.  
🔓 키를 도난당하면 **누구나 암호화/복호화가 가능**하기 때문이다.

---

❓ 그렇다면 **공개 키(비대칭)** 암호화 방식이란?

# 🔓 공개 키 암호화(비대칭) 방식 (Public-key cryptography)

공개 키 암호화는 **두 개의 키**를 사용하는 **비대칭 암호화 방식**이다.

두 개의 키는 다음과 같다:

- **공개 키 (public key)** : 누구나 볼 수 있도록 공개된 키  
- **개인 키 (private key)** : 소유자만 알고 있는 비밀 키

### ⚙️ 작동원리

1. `a`가 `b`에게 데이터를 보낼 때, 공개되어있는 `b`의 **공개 키로 메시지를 암호화**하여 보낸다.
2. `b`는 자신만이 가지고 있는 **개인 키로 해당 데이터를 복호화**한다.

✅ 이 방식은 데이터를 전달하는 과정에서 **중간에서 탈취당하더라도 안전**하다.  
암호화된 정보는 정해진 개인 키 없이 **해독할 수 없기 때문**이다.

---

# 🌐 자, 그럼 HTTPS에서는 어떤 방식으로 데이터를 암호화할까?

➡️ **대칭, 비대칭 방식 두 개 다 쓴다.**

HTTPS는 **SSL/TLS**를 통해 데이터를 암호화한다.

- **SSL (Secure Sockets Layer)** : 넷스케이프가 만든 초기 보안 프로토콜  
- **TLS (Transport Layer Security)** : SSL의 업그레이드 버전이자 현재 표준

> 📝 우리는 흔히 **SSL 인증서**라고 부르지만, 실제로는 대부분 **TLS**를 사용하고 있다.

---

# 📡 HTTPS 접속 시, SSL/TLS가 보안 연결을 설정하는 과정

1. 📨 **클라이언트 연결 요청**  
   클라이언트(브라우저)에서 서버에게  
   `"야! 나 보안 연결 하고 싶어!"` 라고 하며 TLS 버전, 암호 방식 목록을 보낸다.

2. 🏷️ **서버 응답 + 인증서 전달**  
   서버가 **공개 키가 담긴 인증서**를 보냄.  
   이걸로 서버 신원 인증 + 키 교환 준비.

3. 🔐 **클라이언트가 세션 키 생성 & 서버의 공개 키로 암호화하여 전송**  
   👉 이때 사용되는 방식이 **비대칭 키 암호화 방식**

4. 🔓 **서버는 개인 키로 복호화 → 세션 키 확보**

5. ⚡ **이후(TLS 핸드셰이크 이후)는 대칭 키 암호화 방식으로 빠르고 안전하게 통신**

---

📌 이런 식으로 **HTTPS**는 **대칭 + 비대칭 암호화 방식을 혼용**하여  
보안성과 효율성을 모두 챙기는 통신을 수행한다.
